Порт двух готовых React-дэшей (`dashboard.tsx` = Screens A+B, `weight_trajectory_dashboard.jsx`
= Screen C) в один живой Vite-проект, читающий данные из сервиса хранения (ТЗ №2) в рантайме.

Дата сборки: 2026-07-11. Стек: Vite 5 + React 18 + Recharts 2, inline-стили (без Tailwind).

## Слой данных

- Клиент всегда ходит на относительный `/api/labs|weight|dictionary|health`.
  - **dev**: `vite.config.js` proxy → `VITE_API_URL` (по умолчанию `http://localhost:8000`), срезая префикс `/api`.
  - **prod**: `vercel.json` rewrite → `api/proxy.js` (serverless) → `RENDER_API_URL` c заголовком
    `Authorization: Bearer READ_TOKEN`. Токен живёт только на сервере, в клиентский бандл не попадает.
- `useHealthData` при маунте параллельно тянет 4 эндпоинта (`Promise.all`), состояния
  `loading` (скелетоны) / `error` (ретрай-кнопка + сохранение прошлых данных) / `ready`.
- **Кэш в памяти** на модульном уровне (без localStorage/sessionStorage — §5). Кнопка «обновить»
  сбрасывает кэш и перезапрашивает.
- **Мок-режим**: `VITE_USE_MOCK=1` → хук отдаёт `mockData()` вместо fetch. Включён в `.env.development`,
  чтобы дэш открывался без бэкенда. Это dev-инструмент.

## Адаптеры (`adapters.js`)

- `buildMarkers(labs, dictionary)`: группирует `/labs` по `analyte_id` → маркер с
  `points: [{date, ts, value, seq, valueText}]`. Референс берётся из самой свежей строки, где он задан.
  **Обе seq-точки** остаются в ряду (двойные значения видны на графике маркера).
- `computeStatus(direction, value, refLow, refHigh)` — статус на лету (§3.3):
  - `higher_worse`: > refHigh → alert; в пределах BORDER_FRAC ниже refHigh → warn.
  - `lower_worse`: < refLow → alert; в пределах BORDER_FRAC выше refLow → warn.
  - `window`: вне [refLow, refHigh] → alert; в пределах margin (BORDER_FRAC×span) от края → warn.
  - `informational`: без покраски (none).
  - **BORDER_FRAC = 0.08** («граница» = в пределах 8% от порога), конфигурируемая константа в `adapters.js`.
- Статус маркера считается по **последнему** значению; при нескольких `seq` в дату берём `seq=0`
  (проверено: калий с seq0=4.96/seq1=6.5 → статус ok по seq0, обе точки на графике).
- `buildPanels`: маркеры → группы по `panel`, с порядком/русскими именами/типом отображения
  (`PANEL_META`). Тип: `trend` (грид 2×2 sparkline), `serology` (immune_infections → список),
  `reproduction` (карточки по датам). Неизвестные панели сваливаются в `other_markers`.
- `nearestWeight(date, weights)`: ближайший вес + качество (`exact` ±3д, `close` ±30д, `coarse` >30д).
- `pearson(pairs)`: коэффициент корреляции для scatter.

## Screens

- **A (Browse)**: поиск по названию (плоский список результатов), 3 статус-пилюли (фильтр),
  тикер «сейчас вне нормы», список групп с 1–3 горячими отклонениями. Маркер свежести
  `обновлено: <last_ingest>` — в хедере (`App.jsx`).
- **B (Category)**: trend → грид 2×2 карточек-спарклайнов → модалка; serology → список
  «маркер: результат (дата)» с зелёной подсветкой отрицательных; reproduction → карточки
  по датам, раскрываются в таблицу параметров.
- **Модалка маркера**: полный график (обе seq-точки), полоса референса (band / cap / floor
  по наличию границ), опц. оверлей веса (вторая ось, кнопка «вес»), последние замеры.
- **C (Weight)**: траектория веса (все точки из `/weight`, без помесячного усреднения),
  дропдаун оверлея из всех quantitative-маркеров (дефолт LDL), scatter-корреляция с r по
  nearest-date join, точки биоимпеданса (`analyte_id=bia_weight`) как маркеры на линии.
  Веха «старт Cycle IV» (2026-02-01) вертикальной линией.

## Принятые решения (вопросов задать было нельзя)

1. **Одна страница с вкладками, не три экрана-роута.** Хедер общий (маркер свежести + refresh),
   вкладки «Анализы» / «Динамика и связки». Browse↔Category — внутреннее состояние LabsScreen.
2. **Русские имена и порядок панелей** захардкожены в `PANEL_META` (`adapters.js`): API даёт
   `panel` как идентификатор (`lipids_cardio`…), человекочитаемых имён панелей в контрактах нет.
   Порядок — по клинической важности из ТЗ. Новую панель добавить = одна строка в `PANEL_META`.
3. **BORDER_FRAC = 0.08.** ТЗ §3.3 допускает 5–10%, «в пределах 8%» из задачи — взял 8%,
   вынес в константу.
4. **seq=0 для статуса** (не «худший») — как явно указано в задаче; на графике показываем обе точки.
5. **Дизайн-система приведена к индиго-теме ТЗ №4** (`#0a0a1a` фон + радиальное свечение,
   акцент `#818cf8`, янтарь `#fbbf24` только на линии веса Screen C). Исходники были в разных
   тёмных палитрах (dashboard.tsx — тёпло-серая, weight — сине-slate); унифицировал под §4,
   т.к. ТЗ требует единую индиго. Статус-цвета: красный/янтарный/зелёный.
6. **Tailwind-классы исходников заменены на inline-стили** (§5: нет JIT Tailwind). Screen C в
   исходнике был на Tailwind-утилитах — переписан на inline-стили и recharts с токенами темы.
7. **Вес — все точки, без усреднения.** ТЗ разрешает усреднять только если график тормозит;
   100–500 точек recharts тянет без проблем.
8. **Scatter: окно включения ±120 дней** (`MAX_JOIN_DAYS`). ТЗ: «точку без веса в разумном окне
   не включаем». coarse-точки (>30д, но ≤120д) рисуются полупрозрачно; дальше 120д — исключены.
9. **Из Screen C убрал захардкоженные фазы/регрессионные углы/InBody-карточки** исходного
   weight-дэша: они были построены на статичном датасете и наблюдениях, которых нет в live-API
   (нет полей фаз/сегментов). Оставил живое ядро: траектория + оверлей + корреляция + биоимпеданс
   (§3 ac). Веху Cycle IV оставил как единственную конфигурируемую reference-линию.
10. **Биоимпеданс** ожидается как `analyte_id=bia_weight` в панели `body_composition`. Если его
    нет в данных (как в моке) — точки просто не рисуются, без ошибки.
11. **Fragment не передаётся прямым чайлдом графиков** (§5): условные элементы recharts
    (ReferenceArea/Line/ReferenceDot) рендерятся отдельными выражениями, не через `<>`.

# Доработка №2: 8 пунктов после первого просмотра (2026-07-11)

Палитра/контракт API/`api/proxy.js`/`vercel.json` не тронуты. Все правки — inline-стили,

   заменён на симметричный относительно вертикали x=12 канонический контур

   `keydown`; при `Escape` → `onClose()`; снятие в cleanup, зависимость `[onClose]`. Модалка —
   единственный оверлей (в т.ч. открывается из ReproductionView), так что покрыты все кейсы.

3. **Очевидность сброса статус-фильтра.** `LabsScreen.jsx`: под рядом пилюль, когда фильтр
   активен, показывается строка «Показаны только: «<статус>»» + заметный чип-пилюля

   рамка, тень). Повторный клик по пилюле по-прежнему снимает фильтр, но теперь есть явный
   способ сброса.

4. **Вкладка «Вес и динамика» — краш вылечен + страховка.**
   - (а) **ErrorBoundary** (class-компонент в `App.jsx`) оборачивает рендер экранов,
     `key={tab}` (ремаунт при переключении вкладки). При ошибке — «Не удалось отрисовать
     экран: <error>» + кнопка «Перезагрузить». Ошибка логируется в консоль, наружу ничего
     (личные данные).
   - (б) **ПРИЧИНА КРАШ),** найдена воспроизведением (headless jsdom + реальный recharts):
     `<ReferenceLine>` вехи «старт Cycle IV» в `ComposedChart` **не имела `yAxisId`**, а у
     графика оси заданы как `W` и `M` (нет дефолтной `0`). Recharts кидал
     `Invariant failed: Could not find yAxis by id "0"` → весь `WeightScreen` падал → пустой
     экран. Не зависело от данных: вкладка была сломана всегда, а прошлая «проверка» через
     `curl /` возвращала только HTML-оболочку и React в браузере не исполняла. **Фикс:**
     добавлен `yAxisId="W"` к ReferenceLine. Заодно уникализированы ключи
     (`key={ms-${i}}` / `key={bio-${i}}`) — убран варнинг про дубликат ключа `0` между
     MILESTONES и bioDots.
   - **Доп. хардненинг useMemo** (устойчивость к пустым/односерийным/NaN): `weightPts`
     фильтрует строки с невалидной датой/весом; введён `finiteMinMax()` (без spread, игнорит
     NaN) для `wDomain`/`mDomain`/пик/минимум; `mDomain` не вырождается на односерийном
     маркере (min===max → расширение); `yearTicks` строится только при конечных годах и
     диапазоне < 100 лет. Аналогично поправлен `BodyComposition` в `CategoryView.jsx`.

5. **Кнопка «← Назад к группам» крупнее.** `CategoryView.jsx` и drilldown отклонений в
   `LabsScreen.jsx`: padding `9px 16px`→`14px 48px` (шире ≈×3, выше ≈×1.5), fontSize 14→15,

## Экран «Вес и динамика» — два выбираемых ряда A/B (2026-07-11)

Раньше вес был привилегированным основным рядом, менялся только оверлей-маркер.
Теперь **оба ряда выбираемы** и равноправны.

**Что сделано (`src/screens/WeightScreen.jsx`, только этот экран + хелперы в `adapters.js`):**
- Два дропдауна — **«Ряд A» (левая ось)** и **«Ряд B» (правая ось)**. Единый список
  вариантов: `[Вес]` + все quantitative-маркеры (`quantitativeMarkers`). Вес — просто
  один из вариантов (`WEIGHT_ID='weight'`), не привилегированный.
- **Дефолт: A = Вес, B = LDL** (или первый маркер, если LDL нет) → базовый вид как раньше
  (амбер-вес слева + индиго-маркер справа).
- Унификация рядов: `buildSeries(id, role, weightPts, markersMap)` → `{id,name,unit,color,
  dense,points:[{ts,date,value}]}`. `weight` → dense=true, амбер `#fbbf24`, из `data.weight`;
  marker → dense=false, из `markersMap[id]` (value≠null). Цвет: вес всегда амбер; маркер по
  роли — A циан `#22d3ee`, B индиго `#818cf8` (`seriesColor`).
- `ComposedChart`: линия A на `yAxisId="A"` (left), линия B на `yAxisId="B"` (right).
  Плотный ряд — сплошная без точек; разреженный — пунктир с точками (`lineProps` по `dense`).
  `combined` строится по **union ts** обоих рядов; домены осей — `seriesDomain` (устойчив к
  пустому/одноточечному ряду). Тултип/легенда/подписи осей — по именам+единицам выбранных рядов.
- **Scatter A↔B + r пересчитываются** при смене любого ряда. Новый общий
  `joinSeries(sA, sB, maxDays)` в `adapters.js` (+ `nearestPoint`): идёт по более
  **разреженному** ряду, ищет ближайшую точку второго в окне, качество `exact ±3 / close ±30 /
  coarse >30` (coarse полупрозрачно), `MAX_JOIN_DAYS=120`, `r` через `pearson`.
- **bioDots** (биоимпеданс) — только если один из рядов = Вес, привязаны к его оси (`yAxisId`
  ряда-веса).
- **MILESTONES**: у каждого `ReferenceLine` задан `yAxisId="A"` (ось ряда A) — иначе recharts
  падает `Could not find yAxis by id "0"` (та же причина, что ломала экран раньше).
- Вырожденные случаи: A==B → scatter заменяется подсказкой «выберите разные ряды»; ряд с 0/1
  точкой не падает (домены безопасны, pearson n<2 → r=NaN → «—»).
- Приватность: в приват-режиме скрыты числовые подписи осей и тултипы, форма линий остаётся.

**Мок:** `mockData.js` уже содержал и плотный вес (~480 точек), и маркеры разной плотности
(многосерийные + одноточечные `lp_a`/`homa_ir`/`testosterone_free`) — правок не потребовалось.

**Проверки A/B:** `node scripts/validate-jsx.mjs` → 16 ok / 0 failed; `vite build` → зелёная

## Систематизация репродукции и состава тела (2026-07-11)

Обе категории (`reproduction`, `body_composition`) переделаны на **единую снимок-логику**

## Задача A+B — Экспорт PDF в Telegram + округление подписей осей

Два блока в одном заходе. Не тронуты: `api/proxy.js`, `vercel.json`, контракт API, дизайн-система.

### A — Кнопка «⤓ Экспорт PDF в Telegram»

Серверный эндпоинт (через тот же прокси `/api/*`): `GET /api/export/telegram?kind=<analyte|panel>&id=<id>` —
строит PDF (графики→таблицы) и шлёт в Telegram-бот. Ответ `{ ok, sent, filename, markers }` или ошибка (статус ≠2xx).

**Изменённые/новые файлы:**
- `src/api.js` — новая `exportToTelegram(kind, id)`: `fetch('/api/export/telegram?kind=…&id='+encodeURIComponent(id))`,
  возвращает распарсенный JSON; при не-2xx бросает `Error` с текстом из тела (`error`/`detail`/`message`) либо `HTTP <status>`.
- `src/components/TelegramExportButton.jsx` — **новый** переиспользуемый компонент. Один для окна маркера
  (`kind='analyte'`, `id=marker.id`=analyte_id) и шапки категории (`kind='panel'`, `id=panel.id`=слаг панели).
  Состояния: `idle` → `sending` (кнопка disabled, «отправляю…») → `success` («✓ отправлено в Telegram», на 3 с,
  затем обратно в `idle`) или `error` («✕ не удалось» + подпись-подсказка «проверьте связь с ботом и повторите»,
  на 4 с). Стиль акцентный (индиго-пилюля `T.accent`), success/error перекрашивают в зелёный/красный
  (`STATUS_COLORS`). Таймер чистится в `useEffect`-cleanup.
- **Приватность (§A4):** компонент возвращает `null`, когда `usePrivacy() === true` — кнопка **не рендерится**
  вовсе, чтобы не отправить PDF с реальными значениями, показывая дэш кому-то.
- `src/components/MarkerModal.jsx` — кнопка в подвале модалки (под «последние замеры»), `kind="analyte" id={marker.id}`.
- `src/components/CategoryView.jsx` — кнопка в шапке категории (флекс-строка: заголовок+счётчик слева,
  кнопка справа), `kind="panel" id={panel.id}`.

### B — Округление подписей осей до ≤2 знаков

- `src/utils.js` — новая `fmt2(v)`: число → `String(Math.round(v*100)/100)` (≤2 знака, без хвостовых нулей:
  5→«5», 5.4→«5.4», 5.472→«5.47»); не-число возвращает как есть. Отдельно от `formatNum` (тот оставлен без
  изменений — фиксированная точность для карточек/значений).
- `fmt2` подключён как `tickFormatter` ко **всем числовым осям recharts**:
  - `WeightScreen`: обе `YAxis` (A и B) главного графика + обе scatter-оси (X=ряд A, Y=ряд B). `XAxis`-даты не тронуты.
  - `MarkerModal`: `YAxis` значений (L) и `YAxis` веса-оверлея (R). `XAxis`-даты не тронуты.
  - `CategoryView` → `BodyComposition`: `YAxis` веса (тоже числовая ось).
- В тултипах числовые значения прогнаны через `fmt2` (единицы/подписи сохранены): главный график и scatter
  `WeightScreen`, `MarkerModal`, график веса `BodyComposition`.
- `Sparkline` — только линия без числовых подписей → **не тронут** (нечего форматировать).
- **Правило recharts** соблюдено: у каждого `ReferenceLine`/`ReferenceArea`/`ReferenceDot` по-прежнему есть `yAxisId`.
